### 1. UI/UX Redesign (Referencing aura.build)

- **Deep-space gradient background**: `--space-deepest: #030712`, creating an immersive cosmic feel
- **Glass Morphism**: Translucency + backdrop-blur for a frosted glass effect, paired with glow to enhance depth
- **Neon color system**: Blue-purple-magenta gradients (neon blue / purple / magenta), unifying the visual language

- **Canvas particle field effects (ParticleField)**: A Canvas 2D-based particle system rendering starfield / particle connection effects

### 2. 性能优化

- **CSS 动画复杂度降低**：移除 framer-motion 依赖，改用轻量 CSS `anim-up` 类实现入场动画，减少 JS 动画开销
- **engine.ts 提前 emit 与 AnimatePresence 竞态修复**：确保状态更新在动画完成后再提交，避免 UI 撕裂
- **JSON.stringify 快照去重**：在状态变更前做深度比较，相同快照跳过渲染，减少无效 re-render
- **Electron 启动白屏修复**：将 `isDev` 判断逻辑改为 `!app.isPackaged`，确保生产模式正确加载 `file://` 协议

### 2. Performance Optimizations

- **Reduced CSS animation complexity**: Removed the framer-motion dependency and switched to lightweight CSS `anim-up` class for entrance animations, reducing JS animation overhead
- **engine.ts early-emit and AnimatePresence race condition fix**: Ensured state updates are committed only after animations complete, preventing UI tearing
- **JSON.stringify snapshot deduplication**: Performed deep comparison before state changes; identical snapshots skip rendering, reducing unnecessary re-renders
- **Electron startup white-screen fix**: Changed `isDev` logic to `!app.isPackaged`, ensuring correct `file://` protocol loading in production mode

### 3. 关键 Bug 修复

| Bug 描述 | 根因 | 修复方案 |
|---|---|---|
| 新建审查须点两次才能进入 | `TaskNew.tsx` 的 `useEffect` 在组件挂载时无条件执行 `nav()` 跳转 | 引入 `didSubmit` ref，仅在用户实际提交后才触发跳转 |
| 首次点击路由全空白 | React `StrictMode` 双挂载 + `useEffect` 中 `nav()` 污染 HashRouter 内部状态 | 将导航逻辑移至事件回调中触发，避免挂载阶段副作用 |
| Sidebar `layoutId` 消失 | framer-motion 的 `layoutId` 属性在路由切换时丢失引用 | 移除 `layoutId`，改用纯 CSS transition 处理布局动画 |

### 3. Critical Bug Fixes

| Bug Description | Root Cause | Fix |
|---|---|---|
| New review required two clicks to enter | `TaskNew.tsx` `useEffect` unconditionally called `nav()` on mount | Introduced `didSubmit` ref; navigation triggers only after actual user submission |
| First route click rendered entirely blank | React `StrictMode` double-mount + `nav()` in `useEffect` polluted HashRouter internal state | Moved navigation logic into event callbacks, avoiding mount-phase side effects |
| Sidebar `layoutId` disappeared | framer-motion's `layoutId` prop lost its reference during route transitions | Removed `layoutId`, replaced with pure CSS transitions for layout animations |

### 4. 桌面端部署踩坑

- **asar 打包内容**：`electron-main.cjs`、`preload.cjs`、`dist/`、`node_modules/` 必须正确纳入 asar 包，缺一不可
- **node_modules 嵌套结构**：`sql.js` 必须保持 `node_modules/sql.js/` 层级（而非扁平化放到 `node_modules/` 根），否则 `require('sql.js')` 找不到模块
- **isPackaged 判断失效**：部分系统环境下 `app.isPackaged` 返回 `false`，需硬编码 `isDev = false` 强制走 `file://` 协议加载
- **Canvas z-index 遮挡问题**：`position: fixed` 的粒子 Canvas 需确保 `z-index` 低于主体内容层，否则会遮挡所有交互元素

### 4. Desktop Deployment Pitfalls

- **asar package contents**: `electron-main.cjs`, `preload.cjs`, `dist/`, and `node_modules/` must all be correctly included in the asar archive; missing any one will break the app
- **node_modules nested structure**: `sql.js` must retain the `node_modules/sql.js/` hierarchy (not flattened to `node_modules/` root), otherwise `require('sql.js')` cannot resolve the module
- **isPackaged detection failure**: On some systems `app.isPackaged` returns `false`; hardcoding `isDev = false` forces `file://` protocol loading
- **Canvas z-index occlusion**: The `position: fixed` particle canvas must have a `z-index` lower than the main content layer, otherwise it blocks all interactive elements

## 三、技术架构经验

- Electron 生产模式从 `app.asar` 内加载 `file://` 路径的静态资源
- Vite 构建产物为带哈希名的 JS/CSS 文件，`index.html` 自动引用，无需手动管理
- **asar 部署流程**：`vite build` → 提取 asar → 替换 dist 产物 → 重新打包 asar → 替换 `.app` 内 asar → 重启应用
- 开发迭代建议优先用 `vite dev server` 热更新调试，确认无误后再走打包部署流程

## 3. Technical Architecture Insights

- In production mode, Electron loads static assets from within `app.asar` via `file://` paths
- Vite build output consists of hashed JS/CSS files; `index.html` references them automatically — no manual management needed
- **asar deployment workflow**: `vite build` → extract asar → replace dist artifacts → repack asar → replace asar inside `.app` → restart application
- For development iterations, prefer using the Vite dev server with HMR for debugging; only proceed with the packaging/deployment flow once everything is confirmed working

## 7. Technical Debt & Improvement Suggestions

### 1. 构建部署流程正规化
- **现状**：手动 vite build → 提取 asar → 替换文件 → npx asar pack → 替换 .app 内 asar
- **建议**：使用项目中已配置的 electron-builder（`npm run electron:build`），一键完成构建 + 打包 + 签名，避免手动 asar 操作带来的 node_modules 结构错误和 isPackaged 判断问题

### 1. Build & Deployment Process Standardization
- **Current**: Manual vite build → extract asar → replace files → npx asar pack → replace asar inside .app
- **Recommendation**: Use the already-configured electron-builder (`npm run electron:build`) to perform build + package + signing in one step, avoiding node_modules structure errors and isPackaged detection issues caused by manual asar operations

### 2. 粒子场性能优化
- **现状**：100 个粒子 + requestAnimationFrame 全量重绘，在低配设备可能掉帧
- **建议**：
  - 引入 requestAnimationFrame 帧率节流（目标 30fps 而非 60fps）
  - 粒子数量根据设备性能自适应（检测帧率动态调整 PARTICLE_COUNT）
  - 连线检测当前为 O(n²) 复杂度，可考虑空间哈希网格加速

### 2. Particle Field Performance Optimization
- **Current**: 100 particles + requestAnimationFrame full redraw; may drop frames on low-end devices
- **Recommendations**:
  - Introduce requestAnimationFrame frame rate throttling (target 30fps instead of 60fps)
  - Adapt particle count dynamically based on device performance (detect frame rate and adjust PARTICLE_COUNT)
  - Connection detection is currently O(n²); consider spatial hash grid acceleration
