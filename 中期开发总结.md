## 一、引擎核心：动态辩论轮数

### 1.1 问题

系统始终默认为 3 轮辩论，无法根据审查内容的复杂度自适应调整。轮次裁判（round_judge）配置的 phase 被错误归入 `debate_summarizer`，且 system_prompt 为旧版（"基于辩论情况判断"，但实际在准备阶段就被调用，上下文不匹配）。

### 1.2 解决方案演进

| 阶段 | 方案 | 问题 |
|------|------|------|
| v1 | round_judge 调 API → 正则提取轮数 | 提示词与调用时机不匹配，AI 输出不稳定，常 fallback 到 3 |
| v2 | 纯本地规则引擎 `estimateRoundCount()` | 关键词计数粗糙，精度不足 |
| v2.1（当前） | 扩展评分→轮数映射至 1-10 轮，三因子加权判定 | 精度中等，但零延迟零成本，够用 |

### 1.3 当前算法

`estimateRoundCount()` 位于 `src/sdk/engine.ts`，对提取器输出做三因子评分：

- **信号词命中**（8 组正则，分 4/2/1 档权重）
- **维度数量**（结构化标记检测）
- **内容长度**（每 800 字 +1，上限 5）

综合分 → 轮数映射（1-10 轮全量覆盖）。

### 1.4 待定方案 C

已撰写《轮次判定混合方案.md》——规则引擎覆盖极端场景（约 70%），中间模糊地带调一次 round_judge API 兜底，兼顾精度与成本。当前暂不实施，后续视精度需求决定。

## 三、构建与部署修复

### 3.1 应用崩溃（isDev 硬编码）

**根因**：`electron-main.cjs` 中 `const isDev = !app.isPackaged` 在打包后仍返回 `true`，Electron 尝试连接 `http://localhost:5188/` 失败导致白屏。

**修复**：`isDev` 硬编码为 `false`。

### 3.2 tsconfig 路径错误

`config/tsconfig.app.json` 中 `include: ["src"]` 指向不存在的 `config/src/`，应指向 `../src`。修复后 `npm run build` 正常通过。

### 3.3 构建流程

```
源码目录结构：
  config/           ← package.json、vite 配置、ts 配置
  src/              ← 业务代码（engine.ts 等）
  scripts/          ← electron-main.cjs 源文件
  dist/             ← vite 构建产物

构建命令：cd config && npm run build
部署：提取 asar → 替换 dist 产物 → 重新打包 asar
```
