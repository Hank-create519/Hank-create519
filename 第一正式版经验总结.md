# HankAI 审查系统 — 第一正式版经验总结

> 记录第一正式版多轮迭代修复过程中暴露的关键架构问题与解决方案。

---

## 一、skills 工具调用从未生效

### 问题现象
系统看似完整注册了技能 -> 分配了工具 -> 构建了 OpenAI Function Calling 的 tools 参数，但 AI 始终只输出纯文本，从未发起过 tool_calls。日志中 tools 参数明明在请求体里，模型就是不用。

### 根因
数据库配置创建于 skills 功能加入之前。`electron-main.cjs` 的 `seedDefaults()` 中生成了 8 条角色配置，此时 skills 字段尚未成为 seed 逻辑的一部分。后续 skills 功能上线后，数据库已有的旧配置 `skills` 列仍然为空或 `'[]'`。

渲染进程启动时，`App.tsx` 中 `AppInit` 组件从 DB 加载配置。对于从 DB 读出的配置，原逻辑直接使用 DB 中的 skills 值——空数组 `[]`。`engine.ts` 中 `runPhase` 函数据此判断 `hasTools = false`，走 `callAI` 原始路径，tools 参数根本不会出现在请求体中。即使 SkillManager 页面上勾选了技能，`handleApply` 也只是更新了内存中的 `configs` 状态（通过 `useReviewEngine().updateConfigs()`），没有写回 DB。

### 修复方案

**electron-main.cjs（种子修复）**：
- `seedDefaults()` 中每条角色配置追加 `skills` 字段，如 `JSON.stringify(['web_search'])`、`JSON.stringify(['web_search','web_fetch'])`。
- 预置的 `SYSTEM_PROMPTS` 全部补充"强制工具使用规则"段落。

**App.tsx（运行时迁移修复）**：
- `AppInit` 加载 DB 配置后，将其与 `DEFAULT_CONFIGS` 做合并：若配置缺少 `skills` 字段或为空数组，从默认配置回填。
- 将合并后的结果写回数据库（通过 `updateConfig` IPC 持久化），确保后续重启不再丢失。
- 同步修复：若配置的 `systemPrompt` 不包含 `web_search` 关键词，说明是旧版 prompt，也一并回填。

### 教训
- **双层配置一致性是最大坑**：前端 types 中的 `DEFAULT_CONFIGS` 是代码级默认值，electron-main 的 `seedDefaults()` 是数据库种子。两者必须同步维护，否则"新建数据库"和"已有数据库"的行为会不一致。
- **迁移逻辑必须覆盖所有状态**：skills 不仅要在新建数据库时正确写入，还要在已有旧数据库加载时自动补全。App.tsx 中的合并+回写逻辑是第二道防线。
- **内存状态 != 持久化状态**：SkillManager 页面的 UI 操作只改内存，重启就丢。所有用户可见的变更必须有持久化通路。

## 二、systemPrompt 旧版导致 AI 不调用工具

### 问题现象
skills 数组已正确传入、tools 参数已正确构建并随 API 请求发送，但 LLM 仍然不调用任何工具，纯文本回答。

### 根因
OpenAI 兼容的 function calling 机制高度依赖 system prompt 中的引导。模型需要 system message 明确告知"在什么情况下应该调用哪个工具"。旧的 systemPrompt 只有角色职责描述，没有任何工具使用指引。即使 tools 参数在协议层面已正确传递，模型也不知道何时该调用它们。

### 修复方案
- `electron-main.cjs` 中所有角色（extractor、extractor2、debate_ai1/2/3、integrator、final_integrator）的 `SYSTEM_PROMPTS` 全部追加"强制工具使用规则"段落，明确指定何时调用 web_search / web_fetch。
- `App.tsx` 中增加迁移判断：若 DB 中的 systemPrompt 不包含 `web_search` 关键词，说明是旧版本，自动替换为默认版本。
- `src/types/index.ts` 中的 `DEFAULT_SYSTEM_PROMPTS` 同步更新，保持与 electron-main 一致。

### 教训
- **tools 参数只是"允许清单"**，不等于模型会主动使用。system prompt 中的行为引导才是决定 LLM 是否调用工具的开关。
- **Prompt Engineering 是硬依赖**：每新增一个工具，必须在所有相关角色的 systemPrompt 中补充对应的使用场景描述。
- **"能用"和"会用"之间差了一个 system prompt**。

## 三、round_judge 从本地算法改回 AI 判定

### 问题现象
第一版曾实现 `estimateRoundCount()` 本地算法（在 `src/types/index.ts` 中通过正则匹配 `轮数：X` 来判定），任何简单查询都被判为 4 轮辩论，严重浪费 token 和时间。

### 根因
本地正则算法完全无法理解语义复杂度。"今天天气怎么样" 和 "帮我分析公司战略方向" 在算法眼中没有区别——都没有"轮数："关键词，统一 fallback 到 3 轮。而实际上即使是 3 轮对简单问题也太多了。

### 修复方案
- 删除本地 `estimateRoundCount()` 算法（保留 `extractRoundCount` 仅用于显式指定轮数的场景）。
- 在 `engine.ts` 的 `startReview()` 中，Phase 1 完成后，调用 `round_judge` 角色的 AI 裁判（纯 `callAI`，无 tools）判定轮数。
- 裁判输入包含提取器的结构化分析结果，使其能基于内容复杂度做出合理判断。
- 判定结果格式：`"经评估，需要X轮辩论"`，正则提取 X，clamp 到 1~10 范围。
- 裁判调用失败时 fallback 为 3 轮。
- 数据库迁移：`electron-main.cjs` 中将 `round_judge` 的 phase 从 `debate_summarizer` 迁移为 `prep`。

### 教训
- **语义任务不可本地化**：轮数判定本质是语义理解任务，正则/字符串匹配永远无法覆盖真实需求。
- **AI 裁判的成本远低于浪费的辩论轮次**。一次 round_judge 调用约 0.5K token，而多跑一轮 3 个 AI 辩论需要数万 token。
- **fallback 策略要保守**：裁判失败默认 3 轮，而非 1 轮——宁可多跑也不要漏审。

## 四、缺少日期注入导致 AI 误判时间

### 问题现象
`round_judge` AI 裁判判断轮数时，偶尔输出类似"该问题涉及的是 1 年后的事情，建议只进行 1 轮辩论"的判定，导致对实时问题的审查深度不足。

### 根因
LLM 的训练截止日期早于当前日期。当用户问及当年的政策/趋势/数据时，模型基于训练截止日期会误以为"这是未来的事情"，从而调低复杂度评估。

### 修复方案
在 `engine.ts` 的 `startReview()` 中，Phase 1 开始前构造裁判的 user message 时，注入系统当前日期：

```typescript
const todayStr = new Date().toLocaleDateString('zh-CN', {
  year: 'numeric', month: '2-digit', day: '2-digit',
});
const judgeInput = `当前日期：${todayStr}\n\n请基于以下提取器对用户输入的分析结果，判定需要几轮辩论：\n\n...`;
```

### 教训
- **LLM 的时间锚点需要显式注入**。模型不知道"现在"是几点，所有时间相关的判断都必须将当前日期作为上下文的一部分传入。
- **此问题同样适用于所有涉及"最新""当前""今年"等时间敏感词的 prompt**，不仅是 round_judge。

## 五、SkillManager 不持久化

### 问题现象
在 SkillManager 页面分配技能后点击"应用配置"，审查功能正常工作。重启应用后，所有技能分配恢复为初始状态。

### 根因
`SkillManager.tsx` 的 `handleApply()` 函数中：
1. 调用了 `updateConfigs(newConfigs)` —— 只更新内存中的 ReviewEngine 状态。
2. 没有调用 `updateConfig()` 将更改持久化到 SQLite。

而 ReviewEngine 的 `setConfigs()` 函数（`engine.ts`）直接替换 `_state.configs`，没有任何持久化回调。重启后 AppInit 重新从 DB 加载配置，内存中的修改全部丢失。

### 修复方案
在 `handleApply()` 中追加持久化逻辑：遍历更新后的 configs，通过 `window.electronAPI.db.updateConfig()` 逐条写入 DB 的 `skills` 和 `systemPrompt` 字段。失败的角色收集到 `failedRoles` 列表，完成后统一 alert 提示。

### 教训
- **所有用户可见的配置变更必须有持久化通路**。前端状态管理（useState / zustand / engine state）是瞬时的，重启即清零。
- **UI 操作 -> 内存状态 -> DB 持久化**，这三步缺一不可。缺第三步就是"看起来能用但重启就坏的 bug"。
- **批量持久化需要逐条错误处理**：不能一条失败就全回滚（用户已看到部分生效），也不能静默吞掉错误。

## 六、构建路径错误导致黑屏

### 问题现象（反复出现）
`npm run build` 在项目根目录执行时报错 `Cannot resolve entry module ../index.html`。即使通过 `--config config/vite.config.ts` 指定配置文件也无法绕过。Electron 加载 `file://` 协议时出现白屏/黑屏。

### 根因

**第一层：vite.config.ts 的 root 配置**
```typescript
// config/vite.config.ts
export default defineConfig({
  root: '..',  // 关键：root 指向父目录
  base: './',
  build: { outDir: 'dist', assetsDir: 'assets' },
});
```
`root: '..'` 意味着 Vite 的工作根目录是 `config/` 的上级（即项目根目录 `源码/`），`index.html` 在 `源码/` 下。当从项目根目录执行 `vite build` 时，`root` 解析为项目根目录的上级，自然找不到 `index.html`。

**正确构建命令**：必须在 `config/` 目录下执行 `npx vite build`。

**第二层：base 路径问题**
曾将 `base` 设为 `'/'`，Electron 的 `file://` 协议下 `/assets/xxx.js` 会解析为磁盘根目录 `/assets/xxx.js`，导致全部 404，页面黑屏。

**最终配置**：`base: './'`，资源使用相对路径。

### 修复方案
- `vite.config.ts` 中 `base` 固定为 `'./'`，适配 Electron file:// 协议。
- 构建命令必须从 `config/` 目录执行，已在文档/脚本中固化。
- Electron 主进程加载路径：`file://${path.join(__dirname, 'index.html')}`，其中 `__dirname` 指向 `scripts/`，`index.html` 在 `dist/` 下。

### 教训
- **Vite 的 root 配置改变了构建命令的执行位置约束**，不能假设从项目根目录执行一定有效。
- **Electron + file:// 协议下，绝对路径 `/` 指向磁盘根目录**，资源必须使用相对路径。
- **在 README / package.json scripts 中必须固化正确的构建命令路径**，避免每次都要回忆。

## 七、DuckDuckGo 搜索后端不可用

### 问题现象
`web_search` 工具持续超时，返回 "搜索执行失败: DuckDuckGo(timeout)"，AI 模型无法获取实时信息，审查质量严重下降。

### 根因
DuckDuckGo 的 HTML 搜索接口（`html.duckduckgo.com`）在国内网络环境下连接极不稳定，频繁超时（即使 timeout 设为 15 秒）。作为唯一搜索后端时，单点故障直接导致整个 web_search 能力失效。

### 修复方案
在 `src/skills/toolRegistry.ts` 的 `web_search` executor 中实现双后端策略：

1. **Bing 优先**：抓取 `https://www.bing.com/search?q=...` 的 HTML 结果页，解析 `<li class="b_algo">` 块，提取标题和摘要。timeout 12 秒。
2. **DuckDuckGo 兜底**：Bing 失败时自动降级到 DDG HTML 接口，timeout 15 秒。
3. 两者都失败时返回明确的错误信息，包含各自的失败原因。

### 教训
- **搜索后端必须有冗余**。单一后端在国内网络环境下不可靠。
- **Bing 的 HTML 解析比预期稳定**：`b_algo` 类名是 Bing 搜索结果页的长期稳定结构。
- **timeout 设置要分层**：主后端可以设短一些（快速失败），备用后端可以设长一些（给网络更多时间）。

## 八、callAIWithTools 最大迭代不足

### 问题现象
AI 模型在复杂审查任务中进行多轮搜索 + 抓取 + 分析，运行到第 5 轮时抛出"工具循环达到最大迭代次数"，审查中断，输出为空。

### 根因
`callAIWithTools` 的 `MAX_ITERATIONS` 原设为 5。复杂审查任务中，一个典型的工具调用链是：搜索 -> 抓取 2-3 篇文章 -> 基于抓取内容再搜索验证 -> 汇总分析。5 轮迭代在以下场景下极易耗尽：
- 多个 AI 角色共用同一个 session 的工具调用计数
- 单次搜索返回不理想需要重新搜索
- web_fetch 抓取后需要交叉验证

### 修复方案

**扩容**：`MAX_ITERATIONS` 从 5 提升到 15。

**三层保护机制**：

1. **系统提示注入**（第 0 轮）：在首条 system message 后追加提示，告知模型最大轮数，要求尽早输出。

2. **渐进警告**（第 10 轮起）：`WARN_AT = 10`，从第 10 轮开始，每个工具执行结果末尾追加 `[系统提示] 工具调用轮次即将耗尽，剩余 X 轮...`。

3. **强制终止**（第 14 轮，最后一轮）：若模型在第 14 轮仍返回 tool_calls，不再执行工具，而是追加强制终止消息，以不带 tools 参数的方式调用 API，强制模型输出纯文本结论。

### 教训
- **工具循环上限必须根据实际任务复杂度设定**。5 轮在原型阶段看似够用，但真实任务中 AI 会反复搜索验证。
- **渐进式警告比硬截断更优雅**：让模型有时间收尾，而非突然中断。
- **最后一轮的"强制终止 + 无 tools 调用"是兜底关键**：确保即使模型想继续调用工具，也能获得一个文本结论而非空返回。

## 九、多版本代码并存

### 问题现象
项目目录下同时存在桌面版（Electron）和 lobsterai 版两份代码，两个版本的 bug 修复不同步。修复了桌面版的构建问题，lobsterai 版仍然报同样的错。

### 根因
前期为探索不同部署方案，创建了两套独立的代码库。随着桌面版成为主版本，lobsterai 版未及时删除，导致：
- 相似的 bug 需要在两处修复
- 配置文件的差异被遗忘（如 vite.config.ts 的 root/base 设置）
- 新人/新会话可能改错版本

### 修复方案
- 删除冗余的 lobsterai 版本代码。
- 保留唯一的 `源码/` 目录作为唯一代码库。
- 项目根目录只保留文档和 .git。

### 教训
- **多版本并行是技术债**。除非有明确的长期双版本维护计划，否则探索阶段的副本应在确定主方案后立即清理。
- **代码库的"唯一真相来源"原则**：同一项目只应有一个活跃代码库。

## 关键构建注意事项清单

| # | 项目 | 说明 |
|---|------|------|
| 1 | **构建命令** | 从 `config/` 目录执行 `npx vite build`，不可从项目根目录执行 |
| 2 | **base 路径** | `vite.config.ts` 中 `base: './'`，不可改为 `'/'`（file:// 协议兼容） |
| 3 | **Electron 入口** | 主进程从 `scripts/` 加载 `dist/index.html`，`__dirname` 指向 scripts |
| 4 | **DB 种子同步** | 修改 `DEFAULT_CONFIGS`（types/index.ts）必须同步修改 `seedDefaults()`（electron-main.cjs） |
| 5 | **SYSTEM_PROMPTS 同步** | `types/index.ts` 和 `electron-main.cjs` 两份 SYSTEM_PROMPTS 必须保持完全一致 |
| 6 | **Skills 迁移** | 每次新增 skills 字段变更，必须在 `App.tsx` 的 `AppInit` 中追加合并+回写逻辑 |
| 7 | **网络依赖** | `web_search` 依赖 Bing/DDG 双后端，纯离线环境下此功能不可用 |
| 8 | **Python 沙箱** | `python_exec` 在浏览器 dev 模式下不可用，必须在 Electron 环境中测试 |
| 9 | **安全存储** | API Key 通过 Electron safeStorage 加密存储，仅在 Electron 环境下可用 |
